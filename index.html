◆ 核心痛点
自己做 AI 内容创作时发现链路极长——手动刷资讯、筛选题、写文案、做封面，费时费力；直接用 AI 生成又缺少人味，网友一眼能识别。调研 + 访谈后提炼出两个核心痛点：
✕ 资讯获取滞后
一手信息散落在 arXiv、GitHub、X、YouTube 等平台，普通创作者无法每天稳定聚合；依赖二手转述导致内容同质化、时效性差，错过黄金流量窗口
✕ "AI 味"过重
排比句+数字列举、"赋能""颠覆"等惯用表达、所有内容遵循相同模板结构——千篇一律，缺乏个人风格，读者一眼识别
◆ 解决方案：5-Agent 协作工作流
用 CrewAI 框架搭建 5 个专职 Agent 的串行工作流。Agent 2-5 每个节点用户均可介入修改（HITL），不是全程黑盒。
AGENT 01
资讯猎人
全网采集 + 强制溯源一手来源
GLM-4.7
AGENT 02
创意总监
3个差异化选题角度供用户选择
Qwen2.5-7B
AGENT 03
金牌文案
根据选定角度生成正文
Qwen2.5-7B
AGENT 04
视觉策划
同步生成封面 Midjourney Prompt
Qwen2.5-7B
AGENT 05
风格对齐
合规 + 风格一致性 + 质量终检
Qwen2.5-7B
◆ 两个核心机制
HIGHLIGHT
来源约束 + 语义去重
Prompt 硬约束：二手报道强制继续爬取溯源，无一手链接的内容不得写入。ChromaDB 向量检索做历史去重（余弦相似度 > 0.85 判定重复），手动标注 50 条数据遍历阈值选出平衡点。
HIGHLIGHT
偏好对齐增量学习
记录用户在 Agent 3/5 的修改 diff，每 10 条异步触发 LLM 分析，逆向提取高频修改特征（句长、标点、结构偏好），动态重构 System Prompt。不是靠预设模板适配，而是从真实编辑行为里"学"出来的。
◆ 技术决策
DECISION 01
大小模型分发调度
Agent 1 走 GLM-4.7（多工具串行调用需强推理），Agent 2-5 走 Qwen2.5-7B（文本生成靠 Prompt 质量，成本约 1/50）。通过 LLM-as-Judge 横向选型确认。
DECISION 02
多工具编排采集
SerperDevTool 发现入口 → ScrapeWebsiteTool 爬取普通页面 → JinaReaderTool 穿透强反爬（X/Reddit/LinkedIn）→ YouTubeVideoSearchTool 补充视频资讯。
DECISION 03
Prompt Caching
固定 System Prompt + Few-shot 示例标记为可缓存前缀，动态内容放后面。SiliconFlow 兼容 OpenAI 缓存机制，命中后复用 KV Cache 不重复计算。
DECISION 04
语义去重阈值调优
手动标注 50 条历史资讯 ground truth，脚本遍历 0.75-0.95 阈值，观察 precision/recall 变化曲线，选 0.85 作为误杀率和漏判率的平衡点。
◆ 技术架构
CREWAI SEQUENTIAL PIPELINE
用户选题 → 资讯猎人 → 创意总监 → 金牌文案 → 视觉策划 → 风格对齐
STYLE LEARNING LOOP (ASYNC)
用户 diff → SQLite → GLM-4-9b 分析 → 重构 Prompt
SiliconFlow 统一调用，兼容 OpenAI 接口零改造。bge-m3 做向量化，ChromaDB 做语义检索，SQLite 持久化风格偏好。异步分析用 threading.Thread 触发，不阻塞主流程。
